Overview

There are two providers behind one interface, and you choose per request:

  • Single speaker (provider: openai) — OpenAI Whisper. Best for dictation, voice notes, a single narrator.
  • Multiple speakers (provider: assemblyai) — AssemblyAI with speaker labels (diarization). Best for meetings, interviews, calls.

Transcription is asynchronous: you start a job and get a jobId back immediately, then poll until it's done. The same engine powers both the REST API and the MCP tools.

Base URL

https
https://audio-transcriber-ccy7.onrender.com

Authentication

There are no accounts and no app-issued keys. You authenticate with your own provider key, which is also what pays for the work.

  • An OpenAI key (sk-…) for single-speaker transcription and for summaries / action items / translation.
  • An AssemblyAI key for multi-speaker transcription.

Over REST, pass it as a Bearer token. Over MCP, you set it once in the connector config as a header (see below). Keys are held only for the duration of a request and are never logged or stored.

header
Authorization: Bearer YOUR_PROVIDER_KEY

Transcribe a URL

The simplest path: give us a public link to the audio or video and we fetch it.

POST/api/transcribe-url
FieldTypeDescription
audioUrlstringPublic https URL to the file.one of audioUrl / uploadId
uploadIdstringID from an upload slot (see below) instead of a URL.
providerstringopenai (default) or assemblyai.
languagestringISO code like en, he. Omit to auto-detect.optional
keytermsstringComma-separated names / jargon to bias recognition.optional
codeSwitchingbooleanAudio mixes languages. Best with assemblyai.optional
languagesstringComma-separated language names present (hint for code-switching).optional
curl
curl -X POST https://audio-transcriber-ccy7.onrender.com/api/transcribe-url \
  -H "Authorization: Bearer $ASSEMBLYAI_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "audioUrl": "https://example.com/meeting.mp3",
    "provider": "assemblyai"
  }'

# => { "jobId": "b8e0cf4c8f89..." }

You can also send a multipart upload directly to POST /api/transcribe with a file field — that's what the web app uses.

Upload a local file

No public URL? Create a one-time upload slot, PUT the bytes to it, then transcribe by uploadId. This is also how an agent uploads a local file (it can curl -T from its sandbox).

POST/api/uploads
bash
# 1. create a slot
curl -X POST .../api/uploads -H "Content-Type: application/json" \
  -d '{"fileName":"meeting.mp3"}'
# => { "uploadId": "...", "uploadUrl": ".../api/uploads/..." }

# 2. PUT the raw bytes to uploadUrl
curl -T meeting.mp3 "$UPLOAD_URL"

# 3. transcribe by uploadId
curl -X POST .../api/transcribe-url \
  -H "Authorization: Bearer $OPENAI_KEY" \
  -H "Content-Type: application/json" \
  -d '{"uploadId":"...","provider":"openai"}'

Upload slots are single-use and expire after 30 minutes if unused.

Get the result

Poll the job until status is done or error.

GET/api/jobs/:jobId
json
// while processing
{ "status": "processing", "progress": 0.62, "message": "Transcribing…" }

// when done (multi-speaker includes utterances)
{
  "status": "done",
  "transcript": "Full text…",
  "utterances": [
    { "speaker": "A", "text": "Hello.", "start_ms": 0, "end_ms": 900 }
  ],
  "durationSeconds": 612
}

Summary / action items / translation

Run AI post-processing on any transcript. Requires an OpenAI key.

POST/api/post-process
FieldTypeDescription
transcriptstringText to process.required
tasksstring[]Any of summary, actions, translate.required
targetLanguagestringFor translate, e.g. English, Hebrew.optional
modelstringgpt-4o-mini (default), gpt-4o, or gpt-4.1.optional
curl
curl -X POST .../api/post-process \
  -H "Authorization: Bearer $OPENAI_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "transcript": "…",
    "tasks": ["summary", "actions"]
  }'
# => { "summary": "…", "actionItems": ["…"] }

Reattach a job

Multi-speaker work runs on AssemblyAI's servers, so it survives a restart of ours. If you kept the assemblyTranscriptId from a job's status, you can resume it on a fresh server with your AssemblyAI key.

POST/api/assembly/reattach
curl
curl -X POST .../api/assembly/reattach \
  -H "Authorization: Bearer $ASSEMBLYAI_KEY" \
  -H "Content-Type: application/json" \
  -d '{"transcriptId":"…"}'
# => { "jobId": "…" }  then poll /api/jobs/:jobId as usual

Connect to Claude (MCP)

The same engine is exposed as an MCP server, so Claude (or Cursor, or any MCP client) can transcribe and summarize through one connector.

Connector URL

https
https://audio-transcriber-ccy7.onrender.com/mcp

Headers — add your provider key(s) in the connector's custom-header settings. The model never sees them.

headers
X-OpenAI-Key:      sk-…        # single-speaker + summaries
X-AssemblyAI-Key:  …           # multi-speaker (diarization)
In Claude

Settings → Connectors → Add custom connector → paste the URL above, then add the header(s) for whichever provider(s) you'll use. Add only the key for the mode you need.

MCP tools

ToolWhat it does
create_uploadGet an upload URL for a local file. PUT the bytes, then pass the returned uploadId to start_transcription. Skip if you have a public URL.
start_transcriptionStart a job from audioUrl or uploadId. Choose mode: single_speaker or multi_speaker. Returns a jobId.
get_transcriptionFetch a job by jobId. Waits briefly, then returns the transcript (speaker-labeled when available) or asks you to poll again.
post_processSummary, action items, and/or translation for a transcript.

A typical agent flow: start_transcription({ audioUrl, mode: "multi_speaker" }) → get_transcription({ jobId }) (repeat until done) → post_process({ transcript, tasks: ["summary","actions"] }).

Notes & limits

  • Free to use — you pay your provider directly for usage; there's no charge from us.
  • Privacy — audio is processed for the job and the working files are deleted afterward; nothing is retained.
  • Async — always poll /api/jobs/:id (or call get_transcription again). Long files can take several minutes.
  • Code-switching — for audio that mixes languages, use multi-speaker with codeSwitching on; English-plus-one works best. Test a sample first for less common pairs.
  • CORS — the API is callable cross-origin, so browser and server clients both work.
Provider keys are real credentials

Whoever holds a key can spend on that provider account. Use a dedicated key with usage limits if you're wiring this into something shared.
